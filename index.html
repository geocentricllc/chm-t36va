Quick start

        
    
          
  1. Step 1 — Load: drop your .mnt file (from Fusion 360 or EAGLE's mountsmd.ulp) or a CSV onto the dropzone, then Continue.
    2. 
          
  2. Step 2 — Feeder Mapping (optional): for each unique Value+Package, assign a Feeder ID and nozzle. Save the mapping to JSON to reuse on future boards. Skip this step entirely on a first pass and the converter will auto-assign sequential feeder IDs in Step 3.
    3. 
          
  3. Step 3 — Assign Feeders: per-placement table where mapped feeder/nozzle defaults appear. Adjust feeder, nozzle, tape pitch, height, speed, vision per part type; X/Y/rotation per individual placement; tick Skip on any placement to leave it out of the run.
    4. 
          
  4. Step 4 — Board / Calib: set X/Y offset if your file's origin isn't the PCB's lower-left corner. Pick fiducial RefDes from the dropdowns (auto-detected from FID*-style names when present).
    5. 
          
  5. Step 5 — Export: review the DPV preview, click Download. Copy the .dpv file to a USB stick and load on the CHM-T36VA. Calibrate fiducials on the machine before running.
    6. 
        

        
That's the whole flow for a typical board. The rest of this help expands on each piece.


        
Supported file formats


        
Fusion 360 / EAGLE .mnt and .mnb

        
The .mnt / .mnb format comes from EAGLE's classic mountsmd.ulp script, which both EAGLE and Fusion 360's Electronics workspace ship with. Generate it via:

        
    
          
  • Fusion 360 (Electronics): Utilities → Automate → Run ULP → mountsmd.ulp
    • 
          
  • EAGLE: File → Run ULP → mountsmd.ulp from the board editor
    • 
        

        
The script writes one file per side: .mnt = top, .mnb = bottom. The column layout is fixed:

        
RefDes  LocationX  LocationY  Rotation  Value  Package

        
Whitespace-delimited, no header line, comments starting with # / ; / * are stripped. The converter recognizes the extension and skips column-detection automatically.

        
Don't confuse with the Manufacturing → Outputs → Pick and Place export. Fusion 360 has a separate Manufacturing-workspace pick-and-place generator that omits the Package column — that file is incomplete for this converter's needs (the Package field is needed for nozzle/tape-pitch defaults). Use the mountsmd.ulp path instead.


        
CSV / TXT pick-and-place

        
Generic comma/tab/semicolon-delimited files from KiCad, EAGLE, Altium, or Fusion's CAM Processor work too. Column detection is based on header-row name matching:

The first data row is detected automatically — preamble lines like [COMMENT], Centroid Data for…, and [PLACEMENT] are skipped before the header is identified.


        
Coordinate units

        
Auto-detected from the magnitude of the X/Y values:

        
    
          
  • Max coordinate < 13 → inches
    • 
          
  • Max coordinate ≤ 300 → millimeters
    • 
          
  • Max coordinate > 300 → mils (1/1000 inch)
    • 
        

        
The override dropdown on Step 1 lets you change this if the heuristic gets it wrong (rare, but possible for very small boards measured in mils).


        
Feeder mapping

        
Step 2 shows a table with one row per unique Value+Package pair found in your file. For each, you can optionally set a Feeder ID (the station number on your machine where that reel is loaded) and a Nozzle (N1 or N2). Leave blank to skip — those parts will get auto-assigned defaults on the next step.


        
Why this step exists

        
Most physical reels on the CHM-T36VA stay in the same station across multiple boards. Once you've loaded a reel of 10k 0402 at station 5, every future board with that part should use station 5 too. The mapping step lets you record those assignments once and reuse them on every subsequent board via Save Config / Load Config.


        
What the mapping affects

        
On Step 3, every placement of a mapped part type starts with the mapped feeder ID and nozzle as its default. You can still change those defaults on Step 3 — the mapping is a starting point, not a lock. Unmapped part types get the existing auto-guess: a sequential feeder ID and a nozzle picked from the package name (small SMD → N1, larger ICs → N2).


        
Avoiding feeder collisions

        
The Step 3 auto-assigner skips any feeder ID claimed by your Step 2 mappings. So if you map 10k 0402 → 5, no unmapped group will get auto-assigned to feeder 5. This means the duplicate-feeder warning at Step 3→4 only fires for actual conflicts you created intentionally.


        
Save / Load Config

        
The buttons live on Step 2 and operate on the current mapping. Save writes only the rows you actually filled in (blank rows are not stored). Load applies a saved JSON to the current board: any Value+Package matches get the saved feeder/nozzle; unmatched entries are reported in a summary alert.

        
Saved configs are portable across boards: the same mapping works on any future board that includes those parts. Per-placement coordinates and rotations are not in the config — those are board-specific.


        
Workflow walkthrough


        
Step 1 — Load file

        
Drop or click to browse. After parsing, you'll see file stats and an "Auto-detected — override only if wrong" panel with three dropdowns and a one-line summary of which columns mapped to what. Most boards just need Continue.

        
When to touch the overrides:

        
    
          
  • Heuristic picked the wrong units (small mil-units board misread as mm)
    • 
          
  • File has both top and bottom parts and you want both — change Side filter to "All rows"
    • 
          
  • You want to include parts marked DNP — set DNP filter to "Include them"
    • 
        


        
Step 2 — Feeder Mapping (optional)

        
One row per unique Value+Package pair, with two optional inputs: a Feeder ID and a Nozzle toggle. See the Feeder mapping section above for the full rationale.

        
Use the Save config button to write the current mapping to a JSON file. Use Load config to apply a saved mapping to this board (case-insensitive Value+Package match). Clear all wipes the current mapping (useful before loading a different config from scratch).

        
Skip this step entirely on a first pass — the mapping is optional, and unmapped part types just get auto-assigned defaults on Step 3.


        
Step 3 — Assign Feeders

        
The table shows one row per placement on your board. Identical part types share most settings — editing them on one row updates every row of that part type, with sibling inputs visually synced. Editable per row:

        
    
          
  • X / Y — placement coordinates in mm (per-placement)
    • 
          
  • Rotation — placement angle in degrees, CCW positive (per-placement)
    • 
          
  • Skip — flags this individual placement as "don't place" in the DPV (per-placement; ticking on R1 doesn't affect R2)
    • 
          
  • Feeder — station ID where the reel for this part type is loaded (1–99) — shared across same-type parts
    • 
          
  • Nozzle — N1 (left) for small parts, N2 (right) for larger ones — shared
    • 
          
  • Tape pitch — physical reel spacing in mm; see tape pitch reference — shared
    • 
          
  • Height — physical thickness of the part (mm) — shared
    • 
          
  • Speed — 0 means use machine default; raise above 0 only if the part needs slower handling — shared
    • 
          
  • Vision — on for ICs and fine-pitch; can turn off for very small passives — shared
    • 
        

        
Defaults: feeder/nozzle come from your Step 2 mapping if set; otherwise auto-assigned. Tape pitch, height, and nozzle (when not mapped) are guessed from the package name. The auto-assigner skips feeder IDs claimed by mappings, so no collisions sneak in.

        
Click RefDes, Value, or Package column headers to sort. Click again to reverse, third time to return to insertion order. RefDes uses natural sort (R1 < R2 < R10). X / Y / Rotation columns are not sortable. Skipped placements always partition to the end of the table regardless of sort.

        
The table is paged at 20 rows per page. Use Prev / Next or the jump-to-page input. Sorting jumps you to page 1 with the new top of the order. A subtle background tint clusters same-part-type rows visually even when sorted by RefDes.


        
Step 4 — Board / Calib

        
Set the X/Y offset if your file's origin isn't the PCB's lower-left corner; see board origin for details. Pick fiducial RefDes from the dropdowns; see fiducials & calibration.


        
Step 5 — Review & Export

        
The DPV preview shows exactly what will be written. Filename is editable. Download saves a plain-text .dpv file you can copy to a USB stick and load on the CHM-T36VA.

Nozzle selection

        
The CHM-T36VA has two nozzles mounted side by side, named N1 (left) and N2 (right). Each can be a different physical nozzle (CN040, CN065, CN140, etc.) — you switch nozzles on the machine when changing what's mounted.

        
    
          
  • N1 (left) — typically a smaller nozzle for 0201, 0402, 0603, SOT-23. Default for passives.
    • 
          
  • N2 (right) — typically a larger nozzle for SOIC, QFP, QFN, BGA, electrolytics. Default for ICs.
    • 
        

        
The converter auto-picks based on package name. If you only have nozzles loaded for one head and want everything on N1, edit each row or sort by Package and bulk-edit (one row at a time — bulk editing isn't a feature yet).


        
Tape pitch reference

Board origin & rotation

        
The CHM-T36VA places parts using coordinates relative to the PCB's lower-left corner. After fiducial calibration, the machine knows where that corner is in machine coordinates and translates accordingly.

        
Fusion 360 typically exports with the same convention, so X/Y offsets of 0,0 work in most cases.

        
You'll need to set offsets when:

        
    
          
  • Your design uses board-center origin instead of corner origin
    • 
          
  • Negative X or Y values appear in the placements (the machine's origin must be set so all placements are positive)
    • 
          
  • You're panelizing one-up boards into a larger panel and want to shift the placements into the panel's coordinate system
    • 
        

        
Quick test: after Step 3 the table displays placements in the order they'll be made. If the smallest X/Y values are near zero — good. If they're at, say, X = −18 to +18 — the file uses a centered origin and you need an offset of about +18 mm.

        
Global rotation offset (Step 4) rotates every placement by the same angle. Useful if you want to run the same DPV on a board mounted in a different orientation; rare otherwise. For individual reel-vs-footprint orientation mismatches, edit the affected placements' Rotation column directly on Step 3.


        
Fiducials & calibration

        
Fiducials are reference marks on the PCB that the machine's camera uses to figure out exactly where your board is sitting on the bed. The CHM-T36VA expects three:

        
    
          
  • UL — upper-left fiducial
    • 
          
  • LR — lower-right fiducial
    • 
          
  • LL — lower-left fiducial
    • 
        


        
The dropdown picker (Step 4)

        
Each fiducial has its own card on Step 4 with a dropdown that lists every RefDes in your file plus two special options:

        
    
          
  • (none) — leaves X/Y at zero. Use this if your file has no fiducials at all and you'll calibrate entirely on the machine.
    • 
          
  • (custom) — enables manual X/Y entry. Use this if you have fiducial coordinates from another source (e.g. a Gerber file) but they're not in the placement file.
    • 
          
  • Any RefDes — auto-fills the X/Y display from that placement's coordinates, with the board X/Y offset already applied. The X/Y inputs are read-only when a RefDes is selected; selecting a different RefDes or (custom) changes that.
    • 
        


        
Auto-detection

        
If your file has refdes matching FID1 / FID2 / FID3 (also MARK* or F1/F2/F3), the converter pre-selects them as UL/LR/LL on first arrival at Step 4 and shows a small auto-detected badge next to each card's label. The badge disappears once you change the selection.

        
Auto-detection won't override choices you've already made — if you go back and forth between steps, your manual selections stick.


        
Coordinates & the X/Y offset

        
Coordinates flowing into the DPV's CalibPoint section are the placement's raw X/Y plus the board offset above. Changing the offset live-updates the displayed fiducial coordinates so what you see on Step 4 is what gets written.

        
Calibrating on the machine. Even with fiducial coordinates set in the DPV, standard practice is to calibrate at the machine: load the DPV, jog the camera to each physical fiducial mark, and the machine records the actual position. The DPV's fiducial values are a starting hint, not the final word.


        
DPV file format

        
The .dpv file is plain text, CRLF line endings, structured as labeled sections. The CHM-T36VA reads it from a USB stick. Generated files have these sections in order:

        
    
          
  1. separated + header (FILE, PCBFILE, DATE, TIME, PANELYPE)
    2. 
          
  2. Station rows — one per non-skipped part group
    3. 
          
  3. Panel_Array — single board (1×1) by default
    4. 
          
  4. EComponent rows — one per placement (RefDes-level)
    5. 
          
  5. PcbCalib — placeholder (set on machine)
    6. 
          
  6. CalibPoint rows — three fiducial entries
    7. 
          
  7. CalibFator — placeholder (computed by machine)
    8. 
        

        
Coordinates are written with 2 decimal places (the machine truncates further digits anyway). Angles are normalized to (−180°, 180°].

        
Spec quirk: the machine requires at least 2 EComponent rows or it can't compute fiducial calibration. If your job has only one placement, the converter pads with a phantom skipped row to satisfy this requirement.

Saving feeder configs

        
On Step 2, the ↑ Save config button exports your current Value+Package → Feeder/Nozzle mapping to a JSON file. The ↓ Load config button applies a saved JSON to the current board's mapping table.


        
What's stored

        
Only feeder ID and nozzle, keyed by Value+Package. Tape pitch, height, speed, and vision are not saved — they're auto-guessed from package names on every board, which is the right behavior since they depend on the part footprint, not the project. Per-placement coordinates and rotations are also not stored — those are board-specific.


        
How matching works on Load

        
    
          
  • If the new board has a part type with the same Value+Package (case-insensitive), the saved feeder/nozzle is applied to its mapping row.
    • 
          
  • If the new board has a part type not in the config, its mapping row stays blank (and Step 3 will auto-assign a feeder when you continue).
    • 
          
  • If the config has entries for parts not on the new board, they're listed as "unused" in the summary alert that pops up after Load.
    • 
        


        
Old config compatibility

        
Configs saved by older versions of this tool included tape pitch, height, speed, vision, and skip — those fields are silently ignored when loading. The feeder ID and nozzle still apply correctly.


        
Workflow suggestion

        
Tune your feeder assignments carefully on your first board (in Step 2). Save the config. On every subsequent board with the same physical reel layout, load the config first and tweak only the parts you've added. Per-placement rotation tweaks (for reel-vs-footprint orientation fixes) can be applied directly on Step 3 — but those are placement-specific, so if every R1/R2/R3 of a reel needs the same +90° fix, set it once in your CAD library footprint instead so it comes through correctly from the .mnt.


        
Troubleshooting


        
"Couldn't find required column(s)"

        
Auto-detection couldn't identify a required column (RefDes, X, Y, or Rotation) from the header row. Most common causes:

        
    
          
  • File has no header row — open in a text editor and check.
    • 
          
  • Headers use unusual names — supported names are listed in file formats.
    • 
          
  • File has a long preamble that's confusing the header detector. The detector scans the first 15 non-comment lines and picks the most header-like row, but unusual structures can trip it.
    • 
        


        
"After filtering, 0 placements remain"

        
The diagnostic alert breaks down where every row went. Most common causes:

        
    
          
  • Side filter mismatch — your file uses Side values like F.Cu or unusual numbers. Set Side filter to "All rows" or set the Side column to "none".
    • 
          
  • All rows have non-numeric coordinates — file might have units appended ("12.5mm"); the parser strips common unit suffixes but not all.
    • 
        


        
Rotations are off by 90° or 180° on certain parts

        
Almost always a reel-vs-footprint orientation mismatch. Two ways to fix:

        
    
          
  • Best (permanent): fix the footprint in your CAD library so future boards come through correctly.
    • 
          
  • Quick (per-board): sort the table by Value or Package, edit the Rotation column on each affected row to add 90 / 180 / -90 as needed. Tedious for many placements, fine for a few.
    • 
        


        
The machine reports placements off the board

        
Usually a coordinate origin issue. Check Step 4's X/Y offset. If your Fusion file uses board-center origin, the offset should be about (half board width, half board height).


        
"Duplicate feeder IDs assigned"

        
You've assigned the same feeder number to two different part groups. The DPV will still be generated but the machine will physically only have one reel at that station — the second part group will pick the wrong reel. Re-assign one of them to an unused feeder ID.

    • 
          
  • No analytics, no telemetry, no remote calls.
